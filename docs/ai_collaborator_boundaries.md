AI Collaborator Boundaries

Velvet’s public documentation and scaffolding may be drafted, reviewed, or refined with the help of hosted language models.

Those tools are collaborators in text production only.

They are not part of Velvet’s runtime, memory spine, authority chain, policy court, receipt layer, gate, executor layer, or hardware control path.

Boundary Statement

Hosted language models may assist with:

- drafting public documentation
- organizing architecture notes
- reviewing wording for doctrine violations
- generating scaffolding proposals
- formatting project explanations
- helping contributors understand repo boundaries

Hosted language models must not be treated as:

- Velvet’s identity
- Velvet’s memory
- Velvet’s continuity layer
- Velvet’s authority source
- Velvet’s policy court
- Velvet’s receipt chain
- Velvet’s runtime
- Velvet’s executor
- Velvet’s hardware interface

Why This Matters

Velvet is an offline-first machine intelligence framework.

A tool that requires hosted infrastructure cannot be the source of Velvet’s sovereignty. A model that forgets between sessions cannot be Velvet’s continuity spine. A system that cannot produce local receipts cannot be Velvet’s proof layer.

External AI tools may help write about Velvet.

They do not govern Velvet.

Enforcement Principle

No hosted AI collaborator receives runtime authority by contributing documentation, code suggestions, architecture notes, or interface text.

All executable behavior must still pass through Velvet’s normal boundaries:

1. event classification
2. governance policy
3. authorization gate
4. bounded executor or adapter
5. receipt logging
6. state and memory update through approved channels

No proposal becomes action without the gate.

## Runtime Use and Contribution

Hosted AI collaborators may assist during development, documentation, review, and scaffolding.

Once a Velvet system is running, those tools must be treated as external advisory inputs only.

A hosted model may provide:

- suggestions
- summaries
- code review
- documentation drafts
- operator-facing explanations
- non-authoritative analysis
- proposed actions for review

A hosted model must not provide:

- runtime authority
- final authorization
- direct hardware commands
- unreviewed policy changes
- unverified memory writes
- receipt chain authority
- executor access
- bypass paths around Velvet’s gate

If a running Velvet system accepts input from a hosted AI service, that input must enter through the same controlled pathways as any other external source.

It must be classified, checked against policy, authorized through the gate when needed, and recorded by receipts when it affects system state.

Hosted AI output is never trusted because it is fluent.

Hosted AI output is only useful if Velvet can verify, constrain, reject, or receipt it.

## Contribution Boundary While Running

A hosted AI may contribute to the project without becoming part of the machine.

A running Velvet system must remain capable of operating without any hosted model connected.

If a hosted model disappears, resets, changes behavior, loses context, becomes unavailable, or is replaced, Velvet must retain:

- local identity
- local memory
- local policy
- local receipts
- local authority gates
- local executor boundaries
- local operator control

External AI tools may help the human build, inspect, or understand Velvet.

They must not become a dependency for Velvet being Velvet.

Contributor Note

AI-assisted text should be reviewed as draft material.

It may be useful.
It may be wrong.
It may drift from doctrine.
It may lack context.
It is not authoritative until reviewed, accepted, committed, and governed by the project’s normal repository process.

These are not the same thing.
